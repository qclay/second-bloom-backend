# Архитектура и видение проекта (Second Bloom Backend)

Этот документ описывает текущую структуру проекта и основные принципы, чтобы задать единый вектор развития и избежать постоянных концептуальных изменений.

## 1. Общая структура (NestJS)
Проект следует модульной архитектуре. Основные бизнес-сущности разделены по модулям (`src/modules/*`):
- **Auction / Bid:** Логика проведения аукционов, ставок (bids). 
- **Product:** Управление товарами. Товар может быть как обычной покупкой, так и предметом аукциона.
- **Order:** Заказы (покупки товаров или выигранных аукционов).
- **Conversation / Message:** Чаты и переписки. Чаты привязаны к бизнес-контексту (Товар, Аукцион, Заказ, Поддержка).
- **User / Auth:** Пользователи, роли и авторизация.

## 2. Принципы построения API
- **Универсальность эндпоинтов.** Вместо создания множества мелких эндпоинтов под каждую роль (`buyer-chat`, `seller-chat`), мы используем универсальные (например, `POST /conversations/resolve`). Логика внутри сервисов сама определяет права и роли на основе токена (`userId`) и переданного контекста.
- **DTO и Валидация.** Строгая типизация запросов через `class-validator` и генерация Swagger-документации. В ответ всегда отдаем очищенные DTO (например, `BidResponseDto`), чтобы не "сливать" лишние связи из БД фронтенду.

## 3. База данных (Prisma)
- **Soft Delete:** В проекте активно используется механизм мягкого удаления (`deletedAt`), поэтому во всех запросах (особенно сложных `findMany` или `findFirst`) мы обязаны явно проверять `deletedAt: null`.
- **Связи:** Сущности сильно переплетены. Аукцион ссылается на Товар, Ставка (`Bid`) ссылается на Аукцион и Пользователя (`bidder`). При получении данных мы "подтягиваем" (include) нужные связи в зависимости от роли запрашивающего.

## 4. Концепция Чатов (Conversations)
Чаты больше не делятся на "от клиента" или "от продавца". Чат — это сущность, связывающая двух пользователей вокруг какого-то объекта (`Product`, `Order`).
- **Инициация:** Любая сторона (покупатель или продавец) может инициировать чат. Если это делает покупатель, он просто передает ID товара. Если продавец (например, из аукциона) — он передает ID товара и ID потенциального покупателя (`targetUserId`).
- **Безопасность:** На уровне сервиса стоит жесткая проверка: как минимум один из участников чата по товару ДОЛЖЕН быть владельцем этого товара. Пользователям запрещено создавать чаты с самими собой.

## 5. Плановое развитие
- **Фоновые задачи (Cron/Jobs):** Завершение аукционов и деактивация чатов вынесены в `src/jobs`. Это позволяет не блокировать основной поток. В будущем все тяжелые операции (рассылки, пересчет рейтингов) должны проходить через систему очередей (например, Redis/BullMQ).
- **WebSockets / Gateway:** Real-time обновления (новые ставки, новые сообщения) идут через WebSockets. Gateway только транслирует события, вся бизнес-логика и проверки прав должны оставаться внутри Service-слоя.
