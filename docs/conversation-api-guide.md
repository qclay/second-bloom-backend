# Руководство по API чатов для Frontend-разработчика

В рамках рефакторинга мы объединили все сценарии создания/открытия чатов в один универсальный эндпоинт `POST /conversations/resolve`.
Старый эндпоинт `POST /conversations/by-product` удален.

Единый API самостоятельно решает, нужно ли создать новый чат или вернуть уже существующий, основываясь на переданном бизнес-контексте (контекст товара, аукциона, заказа и т.д.). Теперь продавцы могут инициировать чаты с потенциальными покупателями (например, с участниками аукциона), а не только наоборот.

---

## 1. Универсальный эндпоинт

**`POST /conversations/resolve`**

**Требует авторизации:** Да (Bearer Token)

### Формат тела запроса (`ResolveConversationDto`)

```typescript
{
  "context": "PRODUCT" | "AUCTION_BID" | "ORDER" | "SUPPORT",
  
  // ID товара. Обязателен для контекстов PRODUCT и AUCTION_BID
  "productId"?: "uuid", 
  
  // ID заказа. Обязателен для контекста ORDER
  "orderId"?: "uuid",
  
  // ID второго пользователя, с которым мы хотим открыть чат.
  // - Покупатель может не передавать это поле, тогда чат откроется с продавцом товара.
  // - Продавец ДОЛЖЕН передавать это поле (например, ID участника аукциона), чтобы инициировать с ним чат.
  "targetUserId"?: "uuid",
  
  // Любые дополнительные метаданные (необязательно)
  "metadata"?: {}
}
```

## 2. Сценарии использования (Примеры)

### Сценарий А: Покупатель хочет связаться с продавцом по поводу товара (Обычная покупка)

Клиент находится на странице товара и нажимает кнопку "Написать продавцу". Ему не нужно знать `userId` продавца, бэкенд сам найдет его по `productId`.

**Запрос (от лица Покупателя):**
```json
{
  "context": "PRODUCT",
  "productId": "id-товара-123"
}
```

---

### Сценарий Б: Продавец хочет связаться с покупателем/участником аукциона

Продавец зашел в свой аукцион, видит список ставок (bids). В каждой ставке есть данные участника (`bidder.id`). Продавец нажимает кнопку "Связаться" рядом со ставкой.

**Запрос (от лица Продавца):**
```json
{
  "context": "AUCTION_BID", // или "PRODUCT", логика под капотом идентична
  "productId": "id-товара-123",
  "targetUserId": "id-участника-аукциона-456" // Обязательно указываем, с кем именно хотим открыть чат
}
```
*Примечание: Если `targetUserId` не передан, система подумает, что вы хотите открыть чат с продавцом этого товара. А так как вы и есть продавец, API выдаст ошибку `Cannot create conversation with yourself`.*

---

### Сценарий В: Чат по заказу

Когда заказ уже оформлен, и покупатель/продавец хотят перейти в чат заказа. Бэкенд автоматически найдет нужных участников.

**Запрос:**
```json
{
  "context": "ORDER",
  "orderId": "id-заказа-789"
}
```

---

### Сценарий Г: Служба поддержки

Открытие чата с администрацией/саппортом.

**Запрос:**
```json
{
  "context": "SUPPORT"
}
```

---

## 3. Возможные ошибки

- **`400 Bad Request`**: `Cannot create conversation with yourself about this product`. Возникает, если продавец пытается открыть чат по своему товару, но не передал `targetUserId` (и система пытается соединить его с самим собой).
- **`400 Bad Request`**: `One of the conversation participants must be the product seller`. Возникает, если кто-то пытается создать чат по товару, но ни текущий пользователь (`userId` из токена), ни `targetUserId` не являются владельцем (продавцом) этого товара. Чаты "покупатель-покупатель" запрещены.
- **`404 Not Found`**: Товар, заказ или `targetUserId` не найдены в базе.
